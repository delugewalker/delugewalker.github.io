1 版本控制器

版本控制是我们开发软件时的必备工作，将文件的版本进行系统的管理，不仅仅可以帮助我们灵活操作文件，也能简化我们开发的流程。

不使用版本管理器的情况下，如果想要对文件的版本进行管理，需要将不同版本的文件分别存储起来，并分别命名。

随着新版本的迭代，我们的文件则越来越多，也越来越难管理。在团队开发时，如果多人同时对一个文件进行操作，则需要各自保存不同的版本，然后集中审核内容。可想而知，在改动幅度大，人数多的情况下，使用传统方式来开发简直就是噩梦。

而版本控制器就是一个解决这些痛点的神奇软件，它不仅仅可以帮助我们记录每次文件的改动，还可以让同事协同编辑，不仅仅提升了我们个人的开发效率，更简化了团队开发的难度。

市面上曾经出现过很多版本控制器，比如SCCS, RCS, CVS, SVN, BitKeeperSCM，但是都不大好用。在2005年，大牛Linux Torvalds（Linux内核的最早作者）开发出了分布式版本控制器Git，因为它的灵活性和优越性，Git的用户成指数级上升，如今已经发展成了最受欢迎的版本控制器，奠定了版本控制领域的垄断地位。

补充说明：所有的版本控制器系统，只能跟踪文本文件的改动，比如txt文件、网页和所有的程序源代码等等。版本控制器可以追踪每次的改动，比如在第7行添加了新的单词”Git”，在第8行删除了”Max”之类的具体修改信息。但是图片、视频、微软Word这类二进制文件，虽然版本控制器也可以追踪管理，比如从50KB改成了40KB，但是具体改了什么，是没法知道的。

3 Git原理

Git 本地数据管理，大概可以分为三个区，工作区,暂存区和版本库。

  • 工作区(Working Directory)
    是直接编辑的地方，肉眼可见，直接操作。
  • 暂存区(Stage/Index)
    数据暂时存放的区域。
  • 版本库(Commit History)
    存放已经提交的数据，push的时候，就是把这个区的数据push到远程git仓库了。

下面图片可以让我们对Git的操作流程有个简单的理解，一个新建的文件file.txt会在Untracked工作区中，当我们使用git add file.txt后它就会进入Staged暂存区，当我们使用git commit file.txt后文件会进入Unmodified中。

4 Git基本使用

版本库（repository）是一个Git会一直追踪的目录，这个目录中的所有文件的改动，包括各个文件的增加、删除和改动，都会被Git识别出来，以便Git可以追踪目录历史，并在将来某个时刻随时还原。

4.1 配置个人信息(git config)

首先我们先把我们的个人信息告诉git，包括名字和邮箱，这样git就能知道我们的身份，更好地管理每个版本库的贡献者。

$ git config --global user.name "yourname"
$ git config --global user.name "youremail@gmail.com"

我们使用git config加上—global参数，来设置全局配置，这样所有的版本库都会认识我们的身份，当然也可以为特定版本库指定不同的用户名和邮箱(去掉--global)，这里我们直接使用这种省事的方法。

4.2 初始化版本库(git init)

创建一个版本库很简单，第一步先创建一个空目录，然后进入新创建的目录，使用git init命令：

mkdir new_repo
cd new_repo
git init

当然我们也可以进入已存在文件的文件夹，并使用git init命令将其初始化为一个Git版本库。

这样一个Git版本库就创建好了，此时当前目录中还多了一个 .git 目录，这个目录中的信息可以帮助Git来跟踪管理版本库，所以千万不能删除。.git.git.git目录是隐藏目录，需要使用ls -ah命令才能看到。

4.3 初始化版本库(git clone)

使用git clone命令拷贝一个Git仓库到本地，让自己能够查看该项目，或者进行修改。例如在Github上选择一个仓库，点击Code下载按钮，并复制HTTPS的clone链接。

git clone [url]
git clone https://github.com/delugewalker/Linda.git

4.4 添加文件到仓库(git add)

当我们新建文件或修改文件后，需要使用git add命令将其添加到版本库暂存区中。

  • 添加一个或多个文件到暂存区：

    git add [file1] [file2] ...

  • 添加指定目录到暂存区，包括子目录：

    git add [dir]

  • 添加当前目录下的所有文件到暂存区：

    git add .

4.5 提交改动到仓库(git commit)

使用git commit命令将暂存区内容添加到本地仓库中。提交暂存区到本地仓库中:

git commit -m "Create a nre file."
git commit -a

常用Options:

  • -m: 本次提交的说明信息
  • -a: 跳过add到暂存区这一步，直接提交

4.6 查看文件修改状态(git status)

使用git status命令用于查看在你上次提交之后是否有对文件进行再次修改。

git status -s

-s参数表示获取简短输出。

4.7 比较文件差异(git diff)

使用git diff命令比较文件的不同，即比较文件在暂存区和工作区的差异，该命令显示已写入暂存区和已经被修改但尚未写入暂存区文件对区别，有两个主要的应用场景。

  • 尚未缓存的改动(工作区 vs 暂存区)

    git diff

  • 查看已缓存的改动(暂存区 vs 版本库)

    git diff --cached

  • 查看已缓存的与未缓存的所有改动(工作区 vs 版本库)

    git diff head

  • 显示摘要而非整个diff

    git diff  --stat

  • 显示暂存区和工作区的差异

    git diff file.txt

  • 显示暂存区和上一次提交(commit)的差异

    git diff --cached [file]
    git diff --staged [file]

  • 显示两次提交之间的差异

    git diff [first-branch]...[second-branch]

4.8 回退版本(git reset)

git reset命令用于回退版本，可以指定退回某一次提交的版本。也可以用于将添加到暂存区的文件拉回工作区。

  • 取消暂存的文件

    git reset file.txt

  • 重置commit(回退版本)

    • 回退到上一个版本 
      git reset HEAD^
      HEAD^表示回退到上一个版本，^数量表示回退次数
    • 回退到三个版本前 
      git reset HEAD^^^
    • 回退到指定版本 
      git reset 052a
      052a表示commit的hashcode，可以使用git log命令查看。

  • 常见Options

    • –soft: 仅将HEAD指向指定版本的commit，不会修改暂存区和工作区到前一个版本
    • –mixed: 默认为该参数，可以不用带该参数，将HEAD指向以前版本的commit，并重置暂存区的文件与旧版本的提交(commit)保持一致，工作区文件内容保持不变
    • –hard: 将HEAD指向以前版本的commit，同时重置暂存区以及工作区的文件与旧版本的提交(commit)保持一致。这一参数会导致丢失当前工作，如果想要恢复，需要使用git reflog获得原版本的commit hashcode并回溯。

  • reset 与 checkout 的区别
    二者的区别在于 reset 会移动 HEAD 分支的指向，而 checkout 只会移动 HEAD 自身来指向另一个分支。

4.9 删除工作区文件(git rm)

git rm命令用于删除文件。如果只是简单地从工作目录中手动删除文件，运行git status时就会出现Changes not staged for commit的提示。
git rm 删除文件有以下几种形式：

  • 从暂存区和工作区中删除file.txt文件(add&commit) 
    git rm file.txt
  • 如果删除之前已经将file.txt修改过并且放到暂存区域的话(add)，则必须要用强制删除选项-f 
    git rm file.txt
  • 如果想把文件从暂存区域移除，但仍然希望保留在当前工作目录中，换句话说，仅是从跟踪清单中删除，使用--cached 
    git rm --cached file.txt

4.10 移动文件(git mv)

git mv命令用于移动或重命名一个文件、目录或软连接。

  • 移动或重命名文件

    git mv file newfile

  • 如果新但文件名已经存在，但还是要重命名它，可以使用-f参数：

    git mv -f file newfile

5 日志操作

5.1 查看历史提交(git log)

在使用 Git 提交了若干更新之后，又或者克隆了某个项目，想回顾下提交历史，我们可以使用 git log 命令查看。

  • 查看提交历史

    git log

  • 常用options

    • –oneline: 简洁版本

      git log --oneline

    • –graph: 拓扑图

      git log --graph

    • –reverse: 逆向查看

      git log --reverse

    • –author: 指定用户

      git log --author username

    • –since/–before/–until/–after: 指定日期

      git log --before={3.weeks.ago} --after={2010-04-18} --no-merges

  • 退出log界面
    输入q即可退出。

5.2 查看指定文件的修改记录(git blame)

如果要查看指定文件的修改记录可以使用git blame命令，格式如下：

git blame file.txt

查询结果以列表形式呈现。

6 远程管理

6.0 添加远程仓库

以Github为例添加远程代码库，进入Github官网并登录账号。

  • 在设置中找到SSH and GPG keys选项，将本地代码库主机的SSH公钥上传

  • 在Github新建的Repository中点击Code并复制ssh链接

  • 使用该ssh链接即可在本地Git中连接该远程代码库

6.1 远程仓库(git remote)

  • 显示所有远程仓库

    git remote -v

  • 显示某个远程仓库的信息

    git remote show [remotename]

  • 添加远程版本库

    git remote add [remotename] [url]

  • 删除远程仓库

    git remote rm [remotename]

  • 修改仓库名

    git remote rename [old_name] [new_name]

6.2 获取远程代码库(git fetch)

从远端仓库提取数据并尝试合并到当前分支：

git fetch [remotename]
git merge [remotename]/[localbranch]

6.3 获取程代码并合并(git pull)

git pull命令用于从远程获取代码并合并本地的版本。

  • git pull其实就是git fetch和git merge FETCH_HEAD的简写。 命令格式如下：

    git pull [remotename] [remotebranch]:[localbranch]

  • 将远程主机origin的master分支拉取过来，与本地的brantest分支合并

    git pull origin master:brantest

  • (接上一条)如果远程分支是与当前分支合并，则冒号后面的部分可以省略

    git pull origin master

6.4 上传本地分支并合并(git push)

git push命用于从将本地的分支版本上传到远程并合并。

  • 命令格式如下

    git push [remotename] [localbranch]:[remotebranch]

  • 将本地的master分支推送到origin主机的master分支(相同分支可省略冒号及后面)

    git push origin master

  • 如果本地版本与远程版本有差异，但又要强制推送可以使用-force参数

    git push --force origin master

  • 删除远程代码库master分支

    git push origin --delete master

7 Git分支管理

  • 列出分支

    git branch

  • 切换到其他分支

    git checkout branchname

  • 创建并切换到该分支

    git checkout -b newBranch

    与下面指令相同

    1. git branch newBranch
2. git checkout newBranch

  • 删除分支

    git branch -d [branchname]

  • 合并分支

    git merge [branchname]

  • 合并冲突
    手动修改冲突的文件并重新git add该文件

$ 常见问题

1 Git的默认分支

Git软件的默认分支为master，而Github在2020年10月1日将其默认代码库分支从masteri修改为main，为方便起见，我们可以将Git的默认代码库分支做相应的修改。

git config --global init.defaultBranch main

2 忽略某个目录或文件不上传

  • 在代码库中新建.gitignore文件

    touch .gitignore

    该文件可添加忽略上传的规则

  • 修改.gitignore文件，向其中添加规则

    target          //忽略target这个目录
angular.json    //忽略angular.json这个文件
log/*           //忽略log下的所有文件
css/*.css       //忽略css目录下的.css文件

  • 添加并提交

    git add .gitignore
git commit -m "Added .gitignore"
git push -u origin main
